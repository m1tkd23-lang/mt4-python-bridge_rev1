## MVP 完成条件

以下がすべて成立した状態を MVP 完成とする。

---

## 1. 戦略設計とファイル構造

- A戦略（レンジ反転）の **入口・出口・パラメータ** が確定している
- B戦略（トレンド追従）の **入口・出口・パラメータ** が確定している
- A戦略と B戦略は別モジュールで独立実装されている
- A/B それぞれ 3分離構造（_params / _indicators / _rules + 本体）で保守できる
- 最終採用パラメータは **5個以内を目安**
- **SL/TP は戦術ファイル側が正本** (2026-04-21 v2)。combo 戦術は lane 経由で子戦術の値を解決 (`LANE_STRATEGY_MAP` + `risk_config.resolve_lane_risk_pips`)
- `config/app.yaml` の `risk:` セクションは削除。戦術定数必須(未定義なら `SignalEngineError`)

> セクション判定: 暫定達成（A/B ともに入口・出口・パラメータ確定、3分離構造化済、SL/TP 戦術ファイル正本化完了。Walk-Forward 検証後に正式確定予定。2026-04-21 v2）

---

## 2. バックテスト・最適化

- 過去 CSV データに基づくバックテストが実行できる
- 評価期間は **1ヶ月・3ヶ月・半年・1年** の 4 種類で統一されている
- **受入基準: 月別崩壊ゼロ** — 4 期間すべてで月別に分解して崩壊月が出ない戦術のみ採用
- 最悪月劣化度が評価指標として含まれている
- Walk-Forward 検証によるアウトオブサンプル性能確認ができる
- 期間別重み付けスコアと最悪月劣化度が同一レポートで参照できる

> セクション判定: 部分達成(2026-04-21 v3 時点)
> - 月独立 BT: combo_AB +983.2 / 崩壊月ゼロ
> - **連結 BT (12ヶ月連結, 本番相当)**: combo_AB **+1043.0 / 崩壊月ゼロ / worst 2025-06 (-7.0)**
> - 連結 BT は `load_historical_bars_csv_multi` で 65,051 bar を時系列結合して評価、月跨ぎの強制決済/ウォームアップ不連続を排除
> - 未達成項目: (1) 月別崩壊ゼロ判定の自動化、(2) 1ヶ月/3ヶ月/半年の各期間評価、(3) Walk-Forward 検証、(4) スプレッド影響測定、(5) 連結 BT を `run_all_months` 等の標準経路に統合

---

## 3. GUI 運用

- GUI からバックテストを実行できる
- バックテスト結果が可視化される（月別 pips、exit_subtype 分布、SL/TP 発動数、トレード一覧）
- パラメータ探索・精緻化を GUI 上で行える
- 最適化結果を本番パラメータへ反映する導線が **人間承認ゲート経由** で成立している
- 反映はバージョン管理され、ロールバック可能

> セクション判定: 未達（既存 GUI の対策①後動作未確認、承認ゲート未確認）

---

## 4. レジーム判定（A/B 共通の分離モジュール）

- ADX・Efficiency Ratio・BB Width 等でレンジ／トレンドを判定する
- A戦略が機能しない相場環境での **A自動停止判定** に使える
- B戦略の **発動判定** に使える
- A/B どちらの戦術からも差し替え可能な形で実装されている
- 閾値は外部化されており、BT 最適化対象に含められる

> セクション判定: 未達（未実装）

---

## 5. 操作性

- GUI から戦術種別（A / B）とパラメータを指定してバックテストできる
- バックテスト結果が数秒以内に一覧表示される
- 4 期間ロバストネスの合格／不合格が一目で分かる
- 崩壊月（pips が閾値以下の月）が色分け等で明示される
- パラメータ候補→承認→本番反映の導線が破綻なく繋がっている

> セクション判定: 未達

---

## 6. 品質

- A戦略・B戦略・BT基盤のユニットテストが通過する
- 期間別 BT が 1年分 3 分以内で終了する（現在: 約 3 分で全テスト + BT 集計が完了、性能目安は満たす）
- BT 結果が決定論的（同入力で再実行しても同結果）
- MT4 ブリッジは pytest で整合性検証できる
- ログには exit_reason と exit_subtype が両方記録されている

> セクション判定: 部分達成（pytest 15 passed、ログ項目は 2026-04-21 に整備済）

---

## 7. エラー処理・耐障害性

- Python 側クラッシュ時は MT4 側で新規エントリーが停止する
- 想定外状態を検知した場合は新規エントリー停止・既存ポジション維持・人間通知の 3 条件を守る
- 自動リカバリーは行わない
- SL/TP は戦略ロジックで決済されない場合の保険として機能する（現在達成、戦術ファイル定数経由で解決）
- signal_close 比率が月単位で 80% 以上（2026-04-21 v2: B 時刻フィルタ採用で 2026-02 が signal_close 60% → 改善)

> セクション判定: 部分達成（ライブ側フェイルセーフは実装済、signal_close 比率は常時監視必要）

---

## 8. データ整合性

- BT のトレード（ExecutedTrade）に entry/exit 両時点のバンド状態・エントリー理由・エグジット理由・MAE/MFE/保有バー数 が記録されている
- exit_subtype で signal_close の内訳（middle / range_failure / opposite_state / opposite_signal）が追跡できる
- トレードに entry_bar_index / exit_bar_index が記録され、該当期間の decision_logs を紐付けて取り出せる
- CSV/JSONL で外部分析ツールに出力できる

> セクション判定: 達成（2026-04-21 整備完了）
