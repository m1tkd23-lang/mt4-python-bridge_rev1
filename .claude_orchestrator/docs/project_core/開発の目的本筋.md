# 開発の目的本筋

## 1. この開発で作るもの

本開発で作るものは、
**USDJPY の過去5分足月別CSVに対して、ボリンジャーバンド系のA/B 2レーン戦術をバックテスト・可視化・調整し、全月を通して安定利益を狙える最適パラメータと最適なA/B組み合わせを決定するローカル最適化システム** である。

### 主対象

* トレード戦術を設計・検証する開発者（自分自身）
* MT4運用に反映するための戦術コード

### 主な役割

* 月別CSVに対する戦術バックテスト
* A/B各戦術のパラメータ調整
* 月別・全体の成績比較と安定性評価
* GUIによる試行錯誤の高速化
* 最終採択したA/B構成のコード化

### 利用形態

* Windows ローカル環境での Python 実行
* GUI による可視化・調整
* 最終的に MT4 用戦術コードへ反映

---

## 2. 開発の主目的

本開発の目的は、
**data\USDJPY-cd5_20250521_monthly 内の全月CSVを通して、安定して利益を出せるA/B 2レーン戦術のパラメータと組み合わせを確立すること** である。

特に以下を重視する。

* 単月の最大利益ではなく、全月を通した安定性を重視する
* A（レンジ系）とB（トレンド系）の役割分担を維持したまま最適化する
* GUIでのパラメータ調整結果が即時に確認できる状態を作る
* 最終結果が `bollinger_combo_AB.py` に統合可能であることを重視する

### 利益基準

* 月平均で **150〜200 pips 程度の利益**が出ていること
* 全月合算でプラスであること
* 特定月のみで利益を出している構成は採択しない

### 安定性基準

* 赤字月が連続しないこと
* 月別成績のばらつきが極端でないこと
* 最大ドローダウンが破綻レベルに達していないこと

### エントリー基準

* エントリー回数は **100回前後** を目安とする
* ただし回数は絶対条件ではなく、相場状況に応じて増減してよい
* 回数の適正よりも、最終的に安定して利益が出ていることを優先する
* 極端に少ないエントリーや極端に多いエントリーは改善対象とする

---

## 3. MVPの中心機能

以下を end-to-end で成立させる。

### 月別CSVバックテスト機能

* `data\USDJPY-cd5_20250521_monthly` 配下の月別CSVを読み込める
* 単月実行と全月一括実行の両方ができる
* 5分足データを使って戦術A/Bを適用し、トレード結果を再現できる

### 評価・比較機能

* 各月の総pips、勝率、PF、最大DD、取引回数を算出できる
* 全月合算成績を算出できる
* 月別ばらつきや赤字月数を確認できる
* A単体、B単体、A+B 合成成績を比較できる

### GUI最適化支援機能

* `src\backtest_gui.py` を参考に新規GUIを作成する
* GUI上から主要パラメータを変更できる
* パラメータ変更後に再計算し、結果を即時反映できる
* 月別成績表、全体成績、損益推移、必要な補助表示を確認できる

### ポジション管理機能

* 1レーンにつき **常に1ポジションのみ保持可能** とする
* A（レンジ）とB（トレンド）は独立してポジションを持つ
* 同一レーン内での複数ポジションは禁止する
* ポジション状態（未保有 / 保有中 / 決済待ち）を明確に管理する

### ログ・追跡機能

* シミュレーターは「なぜエントリーしたか」「なぜ見送ったか」「なぜ決済したか」を追跡できるログを出力する
* 各トレードは一意な `trade_id` で追跡できること
* 各レーンは `lane_id`（A / B）で区別できること
* ログは人間が読めることに加え、後から集計しやすい構造化形式で保存すること
* シミュレーターと実戦の比較ができるよう、同じ意味体系のイベント名を使うこと

### 最終統合機能

* 評価上位のA/B組み合わせを確定できる
* 採択結果を `src\mt4_bridge\strategies\bollinger_combo_AB.py` に反映できる
* 最終コードで再実行したときに採択時と同等の結果を再現できる

---

## 4. 主線フロー

1. 月別CSVを読み込む
2. A/B戦術と初期パラメータを設定する
3. 単月および全月一括バックテストを実行する
4. 月別成績・全体成績・安定性指標を算出する
5. GUI上でパラメータを変更して再評価する
6. A/Bそれぞれの役割と組み合わせを見直す
7. 最終的に成績の良い2戦術を `bollinger_combo_AB.py` に統合する

---

## 5. 技術方針

* ローカル実行を前提とし、クラウド依存は持たない
* 入力データは過去の月別CSVとする
* GUIとバックテストロジックは分離する
* 評価ロジックは再利用可能な形で独立させる
* 失敗時は処理全体を壊さず、対象ファイル単位で安全に停止・記録する
* `MT4\MQL4` 配下のコードは実運用側の参考資産として扱う

### トレード追跡方針

* 各トレードについて以下を追跡可能にする

  * エントリー理由（どの条件で発火したか）
  * 決済理由（TP / SL / 手動ロジック）
  * 保持中の状態遷移
* ポジションごとに一意IDを持ち、ライフサイクルを追跡できるようにする
* TP / SL の発動タイミングを明確にログ化する

### ログ保存方針

* ログは少なくとも以下の単位で保存する

  * `run` 単位: バックテスト1回全体の情報
  * `bar` 判定単位: 各バーで何を判定したか
  * `trade` 単位: 1トレードの開始から終了までの情報
  * `summary` 単位: 月別および全体集計
* ログ形式は JSON Lines または CSV で機械集計可能にする
* 画面表示用とは別に、元データとして詳細ログを保存する
* 実戦側ログとシミュレーター側ログで `event_type` / `reason_code` / `lane_id` / `trade_id` の概念を一致させる

## 6. 重要な設計方針

* 戦術構成は A/B の2レーンまでとし、3レーン以上には増やさない
* Aはレンジ系、Bはトレンド系という責務を基本維持する
* 初期段階では既存戦術のパラメータ調整を優先する
* 既存戦術で改善不能な場合のみ、必要最小限のロジック追加を行う
* 単月だけ強い戦術は採択しない
* 全月合算成績と月別安定性の両方を採択条件に含める
* GUIは分析・調整支援のための道具であり、採択判断の見える化を担う
* 戦術の複雑化よりも **説明可能性** を優先する

### ログ・分析性

* 「なぜ勝ったか / なぜ負けたか」を説明できるログを必ず出力する
* ログから以下が復元できること

  * エントリー時の市場状態
  * エントリー根拠
  * 決済トリガー
  * 想定シナリオと実際の乖離

### シミュレーターと実戦の一致性

* シミュレーターと実運用（MT4）で挙動差が出ない設計とする
* 以下を一致させる

  * エントリー条件判定
  * SL / TP 処理
  * ポジション管理ロジック
* 「シミュレーターでは勝つが実戦では負ける」状態を防ぐ

### ログフォーマット設計方針

* ログは「説明のため」ではなく「再現と比較のため」に出力する
* 勝った理由、負けた理由を後から分類できる粒度で残す
* 曖昧な自然文だけで終わらせず、必ず `reason_code` を持たせる
* ログの主語を明確にし、`run / lane / trade / bar` のどの単位の出来事かが分かるようにする

### 必須ログ項目

各ログには最低限以下を含める。

* `run_id`
* `strategy_name`
* `lane_id`
* `trade_id`（トレード関連イベント時）
* `event_type`
* `event_time`
* `bar_time`
* `symbol`
* `price`
* `position_side`
* `reason_code`
* `reason_detail`
* `market_state`
* `indicators`
* `position_state_before`
* `position_state_after`

### event_type の標準化

以下のイベント種別を共通利用する。

* `RUN_STARTED`
* `RUN_FINISHED`
* `BAR_EVALUATED`
* `ENTRY_SIGNAL_DETECTED`
* `ENTRY_BLOCKED`
* `ENTRY_EXECUTED`
* `POSITION_HELD`
* `EXIT_SIGNAL_DETECTED`
* `EXIT_EXECUTED`
* `TP_HIT`
* `SL_HIT`
* `FORCE_EXIT`
* `TRADE_CLOSED`
* `ERROR_DETECTED`

### reason_code の標準化

少なくとも以下を使って理由を分類する。

#### エントリー系

* `RANGE_BUY_SIGNAL`
* `RANGE_SELL_SIGNAL`
* `TREND_BUY_SIGNAL`
* `TREND_SELL_SIGNAL`
* `ENTRY_BLOCKED_BY_POSITION_LIMIT`
* `ENTRY_BLOCKED_BY_FILTER`
* `ENTRY_BLOCKED_BY_MARKET_STATE`
* `ENTRY_BLOCKED_BY_EVENT_GUARD`

#### 保持系

* `HOLD_CONTINUE_VALID`
* `HOLD_WAITING_EXIT`
* `HOLD_TP_ACTIVE`
* `HOLD_SL_ACTIVE`

#### 決済系

* `EXIT_BY_TP`
* `EXIT_BY_SL`
* `EXIT_BY_MIDDLE_BAND`
* `EXIT_BY_TREND_REVERSAL`
* `EXIT_BY_MARKET_STATE_CHANGE`
* `EXIT_BY_MANUAL_RULE`

#### 異常・比較系

* `PRICE_GAP_DETECTED`
* `STATE_MISMATCH_DETECTED`
* `SIM_LIVE_DIFF_DETECTED`

### indicators の保存方針

`indicators` には、その判断時点で必要だった数値を保存する。
例:

* `close`
* `ask`
* `bid`
* `bollinger_upper`
* `bollinger_middle`
* `bollinger_lower`
* `trend_ma`
* `trend_slope`
* `band_width`
* `spread`
* `unrealized_pips`
* `sl_price`
* `tp_price`

### 1トレードの追跡要件

1つの `trade_id` について、最低限以下が復元できること。

1. いつ、どのレーンで
2. どの相場状態で
3. どの条件によりエントリーし
4. 保持中に何が起こり
5. どの条件で決済され
6. 最終的に何pipsで終わったか

### 勝因・敗因分析要件

トレード終了時には、`TRADE_CLOSED` イベントとして以下を必ず残す。

* `entry_reason_code`
* `exit_reason_code`
* `holding_bars`
* `max_favorable_excursion_pips`
* `max_adverse_excursion_pips`
* `realized_pips`
* `win_or_loss`
* `outcome_category`

`outcome_category` 例:

* `GOOD_RANGE_CAPTURE`
* `GOOD_TREND_CAPTURE`
* `SMALL_STOP_LOSS`
* `LARGE_STOP_LOSS`
* `LATE_EXIT`
* `EARLY_EXIT`
* `FALSE_ENTRY`
* `MISREAD_MARKET_STATE`

## 7. 制約条件

### 技術制約

* Pythonローカル環境で動作すること
* ベース戦術は `src\mt4_bridge\strategies\` 以下の既存構成を利用すること

### 運用制約

* A/Bの2レーン構成を維持すること
* 各レーンは1ポジション制約を厳守すること
* 既存の `bollinger_range_v4_4.py` と `bollinger_trend_B.py` は出発点であり固定ではない
* 必要なら新パラメータや派生戦術を追加してよいが、レーン数は増やさない

### 性能制約

* GUI上でのパラメータ変更に対し、実用的な速度で再評価結果が見えること
* 複数月一括評価が現実的な時間で完了すること

### ログ品質制約

* ログが取れないロジックは採択しない
* ログは後で月別・レーン別・`reason_code` 別に集計できる形式であること
* 自然文だけのログは禁止し、構造化項目を必須とする
* 勝敗の原因を分類できないログは不十分とみなす
* シミュレーター専用の曖昧な簡略判定は避け、実戦側へ移植可能な設計にする

### 環境制約

* Windows環境を前提とする
* 対象データは `data\USDJPY-cd5_20250521_monthly` 配下のCSVとする
