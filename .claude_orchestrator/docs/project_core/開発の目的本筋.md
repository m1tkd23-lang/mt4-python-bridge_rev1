# 開発の目的本筋

最終更新日: 2026-04-21

---

## 1. この開発で作るもの

本開発で作るものは、  
**ボリンジャーバンドを基礎とした USDJPY 5分足の自動売買システム（MT4 + Python 連携型）** である。

### 主対象
- USDJPY 5分足の値動き
- MT4 上で稼働する自動売買ボット
- バックテスト・最適化・運用を一貫して行うデスクトップアプリケーション

### 主な役割
- MT4 から受けた相場情報を基に Python 側でシグナル生成・判断を行う
- レンジ相場・トレンド相場の判別に基づいた適応的なエントリー判断
- 過去データに基づく戦略パラメータの最適化と検証
- ライブ運用時のポジション管理・トレードログ記録

### 利用形態
- ローカル環境（Windows + MT4 + Python）での常時稼働
- 開発者本人による個人運用（自己資金での少額検証から開始）
- GUI アプリで最適化と運用を操作
- MT4 は薄いブリッジに徹し、判断ロジックは全て Python 側に集約する

---

## 2. 開発の主目的

本開発の目的は、  
**統計的・システマチックなアプローチで再現性のあるトレード戦略を構築し、長期にわたり市場で退場しない自動売買基盤を成立させること** である。

特に以下を重視する。

- **退場しないこと**  
  勝つことよりも、破綻しないことを最優先とする。
- **再現性のある検証**  
  思いつきではなく、バックテスト・Walk-Forward 検証に基づいて戦略を構築する。
- **過学習への抵抗力**  
  直近のみで機能する戦略を排除し、複数期間でロバストな戦略を採用する。
- **運用・改善サイクルの高速化**  
  GUI による最適化結果の確認と、検証ゲートを経たパラメータ反映を行えるようにする。

## 3. MVPの中心機能

当面の主線は **BT 基盤と GUI 反映** である。ライブ運用は後段とする。

### 3.1 戦略設計とファイル構造
- A戦略（レンジ反転） / B戦略（トレンド追従）それぞれの **入口・出口・パラメータ** を確定させる
- A戦略と B戦略は別モジュールで独立実装する（レジーム判定器・リスク管理・シグナルエンジンは戦略から分離）
- B戦略も将来の本番稼働を見越してファイルを揃えるが、まずは正しくバックテストすることを優先する

### 3.2 バックテスト・最適化
- 過去 CSV データに基づくバックテスト実行
- 評価期間は **1ヶ月・3ヶ月・半年・1年** の 4 種類で統一する
- **受入基準: 月別崩壊ゼロ**
  - 崩壊月とは、月次損益・最大ドローダウン・PF・取引件数などが許容基準を著しく下回る月を指す
  - 具体的な閾値は評価器仕様で別途定義する
- Walk-Forward 検証によるアウトオブサンプル性能確認
- 期間別重み付けスコアに加え、**最悪月の劣化度** も評価指標に含める

### 3.3 GUI 運用
- バックテスト実行と結果可視化
- パラメータ探索・精緻化
- 最適化結果から本番パラメータへの反映（人間承認ゲート経由）

### 3.4 レジーム判定（A/B 共通の分離モジュール）
- ADX、Efficiency Ratio、BB Width 等によるレンジ／トレンド判定
- A戦略の停止判定および B戦略の発動判定に利用する
- 戦略ロジックとは分離し、差し替え可能な形で実装する

### 3.5 ライブ運用（後段）
本節は当面の主線外。BT 基盤と GUI 反映が整った後に着手する。

- MT4 から受けた価格情報に基づく Python 側でのシグナル生成
- MT4 への売買指示出力とポジション状態の整合性検証
- ハートビート・Stale 検知によるフェイルセーフ
- ライブ状態の監視（GUI）

## 4. 主線フロー

### 4.1 当面の主線（BT 基盤と GUI 反映）

1. 過去 CSV データを読み込む  
2. 特徴量計算およびレジーム判定を実施する  
3. A戦略・B戦略をそれぞれバックテストする  
4. 月別分解・複数期間評価・Walk-Forward 検証を行う  
5. 評価スコアおよび崩壊月の有無を判定する  
6. 候補パラメータを GUI に表示する  
7. 人間が確認し、承認した場合のみ本番パラメータへ反映する  

### 4.2 ライブ運用フロー（後段）

1. MT4 が価格情報（ティック・バー確定）を Python ブリッジに受け渡す  
2. Python 側で特徴量計算・レジーム判定を実施する  
3. 戦略条件を満たした場合にシグナルを生成する  
4. リスク管理を経て MT4 に売買指示を出力する  
5. 約定・決済結果を受け取りログに記録する

## 5. 技術方針

- **ローカル完結**  
  全機能がローカル Windows 環境で完結する。外部サーバー・クラウド依存を持たない。

- **データ保持**  
  当面は CSV ベースとする。将来的に SQLite または DuckDB への移行を計画する。

- **外部依存の扱い**  
  MT4 連携は既存のファイル／ブリッジ方式で維持する。ブローカー API への直接接続は行わない。

- **失敗時の挙動**  
  Python 側クラッシュ時は MT4 側で新規エントリー停止とする。  
  想定外の状態を検知した場合はエントリーを停止し、既存ポジションは維持する。  
  不整合検知時は人間に通知する。自動リカバリーは行わない。

- **拡張性・分離性**  
  A戦略と B戦略は別モジュールで独立実装する。  
  レジーム判定器・リスク管理・シグナルエンジンは戦略から分離し、差し替え可能とする。

---

## 6. 重要な設計方針

- **MT4 は薄く、Python が判断する**  
  売買判断ロジックはすべて Python 側に集約する。

- **検証ゲートによるパラメータ反映**  
  候補 → 検証 → 人間承認 → 反映 → バージョン管理 → ロールバック可能  
  の段階を必須とする。

- **A戦術・B戦術の両方を磨き込む（ただし主線はBT）**  
  A戦略 / B戦略それぞれを独立に設計・BT・評価する。  
  当面は **BT 基盤と GUI 反映を主線** とし、ライブ本番稼働は後段とする。

- **「崩壊月ゼロ」を採用基準とする**  
  全評価期間において、月別に分解したときに崩壊月が存在しない戦術のみ採用する。

- **過学習対策を最上位に置く**  
  必ず複数期間評価および Walk-Forward 検証を通す。

- **ルール通りに負けるを許容する**  
  想定内の損失は戦略の正常動作とみなし、後付け調整を抑制する。

- **バー確定後に判断する**  
  未確定バーは使用しない。

## 7. 制約条件

- **技術制約**  
  Windows・MT4・Python 前提。MT5 等には対応しない。

- **運用制約**  
  個人開発・個人運用。短時間で判断可能な構造を優先。

- **性能制約**  
  5分足で数秒以内の処理完了。

- **環境制約**  
  スプレッドは固定値とせず、可変パラメータとして扱う。  
  バックテスト時には複数条件（例: 0.0 / 0.2 / 0.5 pips）で検証を行い、スプレッド変動に対する耐性を評価する。

- **資金制約**  
  生活に影響しない範囲で運用。
