# task_splitting_rules

## 目的

この文書は、task_router が task を安全な粒度に整理するための分解ルールを定義する。  
大きすぎる task、曖昧すぎる task、複数責務が混ざった task を避けるために使う。

---

## 基本方針

- 1task 1責務を基本とする
- 実装と検証を一度に抱え込みすぎない
- 大きい変更は段階的に分ける
- UI 変更と内部ロジック変更は、必要なら分割する
- 「とりあえず全部直す」は禁止
- 人や後続 role が完了判定しやすい粒度を優先する

---

## 良い task 粒度

以下のような状態を目指す。

- 変更対象が明確
- 完了条件が明確
- context_files が最小限で足りる
- implementer が安全に着手できる
- reviewer が確認しやすい
- director が approve / revise / stop を判断しやすい

---

## 分割を検討すべきパターン

### 1. 複数責務が混ざっている
例:
- usecase 追加
- GUI 接続
- tests 追加
- docs 更新

が同時に入っている場合。

対応:
- まず内部ロジック
- 次に GUI 接続
- 次に統合確認や docs
のように分けることを検討する。

### 2. 変更対象が広すぎる
例:
- 「全ての XX を修正する」
- 「repo 全体を整理する」
- 「UI を全部改善する」

対応:
- 対象画面
- 対象 usecase
- 対象機能
で区切る。

### 3. 成功条件が曖昧
例:
- 「使いやすくする」
- 「改善する」
- 「最適化する」

対応:
- 完了条件を具体化できなければ blocked を検討する。

### 4. 前提不足
例:
- 必要ファイルが不明
- 仕様が未確定
- completion_definition や feature_inventory と接続しない

対応:
- blocked
- 事前整理 task
- 調査 task
への分割を検討する。

---

## UI / 内部ロジック分割ルール

### 同一 task に含めてよい場合
- 変更対象が狭い
- 内部ロジック追加と GUI 接続が一対一で自然に結びつく
- reviewer が無理なく確認できる
- tests や validation も局所で閉じる

### 分けた方がよい場合
- 内部ロジック追加が大きい
- GUI 変更範囲が広い
- 先に usecase / infrastructure を固めた方が安全
- 複数画面や複数フローに波及する
- Excel / CSV / 印刷など外部形式連携を伴う

---

## 統合確認 task を切るべき場合

以下の場合は、個別実装 task の後に統合確認 task を検討する。

- 複数 task にまたがる機能が finally つながる
- GUI と usecase と import/export がまとめて成立する必要がある
- end-to-end 動作確認が価値を持つ
- completion_definition 上の主要項目を閉じる最終確認が必要

---

## blocked を検討すべき場合

- 粒度調整では解決できない曖昧さがある
- 成功条件が定義できない
- 実在確認できない前提が多い
- 完了意図と禁止事項が両立しない
- どこで分けても安全に着手できない

---

## task_router 用判断ルール

task_router は以下を意識すること。

- 大きすぎる task をそのまま ready にしない
- completion_definition に対して、どの不足を埋める task かを明確にする
- feature_inventory に照らして、新規実装か GUI 接続か統合確認かを見分ける
- 必要なら「先に usecase」「次に GUI」「最後に統合確認」といった段階分けを提案する

---

## 更新ルール

- repo の進行に応じて見直してよい
- 完成定義や feature_inventory と矛盾しないように保つ
- 抽象論ではなく、実際の task 分解に使える粒度で保つ
