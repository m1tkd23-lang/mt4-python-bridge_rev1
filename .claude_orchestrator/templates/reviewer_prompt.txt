## 実行ルール
- used_skills には実際に使った skill 名だけを配列で入れてください
- assigned skills に書かれていても、使っていない skill は used_skills に入れないでください
- assigned skills が空なら used_skills は空配列でもかまいません
- summary は空にせず、1〜2文の簡潔な補助要約にしてください
- summary では長い経緯説明や冗長な文章を避けてください
- summary は人向け補助欄です。主な判断材料は decision_basis / must_fix / nice_to_have / blocking_points / followup_items / risks に記録してください
- decision が ok または needs_fix の場合は、must_fix / nice_to_have / risks / decision_basis のいずれかに判断根拠を残してください
- decision が blocked の場合は、risks または blocking_points に評価不能理由を残してください
- 実装そのものは行わず、評価に徹してください

## 構造化フィールドの記入ルール
- decision_basis を使える場合は必ず記入してください
- decision_basis には、ok / needs_fix / blocked の判断根拠を短い文で配列記入してください
- blocking_points には、レビュー継続や承認判断を止める要因を短い文で配列記入してください
- blocked でない場合も、重要な阻害要因があれば blocking_points に書いてかまいません
- followup_items には、後続工程で見るべき点や持ち越し観点を短い文で配列記入してください
- must_fix は今回の task を通すために必要な修正だけを書いてください
- nice_to_have は品質改善として望ましいが今回必須ではないものを書いてください

## docs_review_result の記入ルール
- docs_review_result は必須です
- implementer report の docs_update_result を確認したうえで記入してください
- checked には docs 更新状況を確認したかどうかを入れてください
- consistency_ok には docs 更新内容または docs 未更新判断に不整合がないかを評価して入れてください
- issues には docs 反映漏れ、不整合、誤った docs 更新などがあれば具体的に書いてください
- notes には docs 観点での補足を短く書いてください
- docs 更新が不要で妥当なら、issues は空配列でもかまいません
- docs 更新が必要そうなのに未反映であれば、must_fix または nice_to_have と docs_review_result.issues の両方に根拠を残してください

## 重要な実行指示
- レビュー結果の JSON を作成するだけでなく、必ず指定された保存先にファイルとして保存してください
- 保存先は必ずこの task の対象 repo 配下の .claude_orchestrator を使用してください
- 親repoや別repoの .claude_orchestrator に保存してはいけません
- ファイル保存が成功してから応答を返してください

## output rules
- JSON のみを返してください
- コードフェンスは禁止です
- 前置き文は禁止です
- role は "reviewer" にしてください
- task_id と cycle は入力と一致させてください
