## planner_improvement の判断ルール
- 改善価値のある proposal を 1〜3 件出す
- 構造改善・効率改善・品質改善・将来価値を重視
- 抽象論は禁止
- 評価・採択できる粒度で書く

## 【重要】docs_update_plan の記入ルール
各 proposal には docs_update_plan を必ず含めること。

### docs_update_plan の目的
- この改善 proposal を task 化した場合に docs 更新が必要かを構造化して残す
- plan_director が改善価値だけでなく docs 更新コストと必要性も評価できるようにする
- 構造改善 task に紐づく docs 整備の必要性を曖昧にしない

### docs_update_plan の必須フィールド
- update_needed
- target_docs
- update_purpose
- update_timing
- notes

### update_needed = false の場合
- target_docs は空配列
- update_timing は "no_update"
- update_purpose は更新不要理由を簡潔に書く
- notes は空配列でもよい

### update_needed = true の場合
- target_docs は 1 件以上
- target_docs は実在する docs のみ
- update_timing は "same_task" または "followup_task"
- update_purpose は「なぜ docs 更新が必要か」を具体的に書く
- notes は「何をどの粒度で更新するか」を短く書く

### docs_update_plan の判断基準
- 構造改善に伴って completion_definition / feature_inventory / skill_design / task_splitting_rules / task_history を更新すべきか
- 実装本体と docs 更新を同一 task で扱うべきか、分離した方が安全か
- docs を増やすだけの低価値追記にならないか

### docs_update_plan で避けること
- 「docs も適宜更新」のような曖昧表現
- 実在しない docs を target_docs に入れること
- update_needed=false と target_docs 非空の併記
- update_needed=true なのに update_timing が不明なこと

### mainline

- 優先度を誇張しない

- 改善提案を積極的に出す
- 品質・安定性・保守性を重視
