## 実行ルール
- used_skills には実際に使った skill 名だけを配列で入れてください
- assigned skills に書かれていても、使っていない skill は used_skills に入れないでください
- assigned skills が空なら used_skills は空配列でもかまいません
- summary は空にせず、1〜2文の簡潔な補助要約にしてください
- summary では長い経緯説明や冗長な文章を避けてください
- summary は人向け補助欄です。主な判断材料は changed_files / commands_run / results / completion_level / unresolved_items / next_task_hints に記録してください
- status が done の場合は、少なくとも changed_files / commands_run / results のいずれかに実行証跡を残してください
- status が blocked または need_input の場合は、risks または questions に停止理由や確認事項を残してください
- 実在しないファイルや実行していないコマンドを記録しないでください

## 構造化フィールドの記入ルール
- completion_level を使える場合は必ず記入してください
  - full: 今回 task で期待された主要作業が完了した
  - partial: 一部完了だが未解決項目が残る
  - minimal: 最小限の確認や部分作業のみで、実質未完了に近い
- unresolved_items には、今回解消できなかった項目を短い文で配列記入してください
- unresolved_items には「何が」「なぜ残ったか」が分かる表現を入れてください
- next_task_hints には、後続 task の候補や持ち越し観点を短い文で配列記入してください
- next_task_hints は「次に何を task 化すべきか」が分かる表現にしてください
- results は人向け長文説明ではなく、実行結果・証跡・検証結果を優先して記録してください

## docs_update_result の記入ルール
- docs_update_result は必須です
- この task 実行中に docs を実際に更新したかどうかを記録してください
- docs を更新した場合は update_performed = true にし、updated_docs に実際に更新した docs パスを入れてください
- docs を更新していない場合は update_performed = false にし、updated_docs は空配列にしてください
- update_summary には「何を更新したか」または「なぜ更新しなかったか」を簡潔に書いてください
- pending_followup_docs には、今回更新しなかったが後続で更新が必要そうな docs を配列で書いてください
- 実在しない docs を updated_docs / pending_followup_docs に入れてはいけません
- docs 更新不要なら、その理由が update_summary から読めるようにしてください

## 重要な実行指示
- 作業結果の JSON を作成するだけでなく、必ず指定された保存先にファイルとして保存してください
- 保存先は必ずこの task の対象 repo 配下の .claude_orchestrator を使用してください
- 親repoや別repoの .claude_orchestrator に保存してはいけません
- ファイル保存が成功してから応答を返してください

## output rules
- JSON のみを返してください
- コードフェンスは禁止です
- 前置き文は禁止です
- role は "implementer" にしてください
- task_id と cycle は入力と一致させてください
