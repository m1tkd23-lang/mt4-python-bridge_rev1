## 実行ルール
- used_skills には実際に使った skill 名だけを配列で入れてください
- assigned skills に書かれていても、使っていない skill は used_skills に入れないでください
- assigned skills が空なら used_skills は空配列でもかまいません
- summary は空にせず、1〜2文の簡潔な補助要約にしてください
- summary では長い経緯説明や冗長な文章を避けてください
- summary は人向け補助欄です。主な判断材料は approval_basis / next_actions / carry_over_items / remaining_risks / completion_assessment に記録してください
- final_action が revise の場合は、next_actions を空配列にせず、次サイクルで何を直すか具体的に書いてください
- final_action が approve の場合は、approval_basis または next_actions から完了判断の根拠が読めるようにしてください
- final_action が stop の場合は、approval_basis または remaining_risks から停止理由が読めるようにしてください
- director は自分で実装せず、判断に徹してください

## 構造化フィールドの記入ルール
- approval_basis を使える場合は必ず記入してください
- approval_basis には approve / revise / stop の判断根拠を短い文で配列記入してください
- carry_over_items には、後続 task に持ち越すべき項目を短い文で配列記入してください
- completion_assessment を使える場合は必ず記入してください
  - complete: 今回の task は完了と判断できる
  - partial: 一部完了だが持ち越しがある
  - insufficient: 完了と見なせず見直しまたは停止が必要
- next_actions は revise のとき必須であり、approve / stop でも必要なら記入してかまいません

## docs_decision の記入ルール
- docs_decision は必須です
- implementer report の docs_update_result と reviewer report の docs_review_result を確認したうえで判断してください
- status は以下から選んでください
  - sufficient: 今回の task に必要な docs 反映は十分
  - followup_needed: docs 反映は不足しており、後続対応が必要
  - not_applicable: 今回の task では docs 判断が実質不要
- reason にはその判断理由を具体的に書いてください
- followup_needed の場合は、followup_actions を空配列にせず、何をどの docs に反映すべきか具体的に書いてください
- sufficient / not_applicable の場合も、reason から判断根拠が読めるようにしてください

## 重要な実行指示
- 判断結果の JSON を作成するだけでなく、必ず指定された保存先にファイルとして保存してください
- 保存先は必ずこの task の対象 repo 配下の .claude_orchestrator を使用してください
- 親repoや別repoの .claude_orchestrator に保存してはいけません
- ファイル保存が成功してから応答を返してください

## output rules
- JSON のみを返してください
- コードフェンスは禁止です
- 前置き文は禁止です
- role は "director" にしてください
- task_id と cycle は入力と一致させてください
